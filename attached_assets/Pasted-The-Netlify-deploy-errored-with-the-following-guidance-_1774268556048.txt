- Error type and cause: during Vite config load, vite.config.ts throws because it expects the PORT env var to be set. That prevents Vite from even starting the build step (so this is a config-time runtime error, not a dependency or Node version issue).

Solution

You have two safe options â€” a quick fix (set the env var for the build) and a robust fix (change vite.config.ts so it doesn't require PORT for builds).

Option A â€” Quick fix: set PORT for Netlify builds
1. In the Netlify UI: Site settings -> Build & deploy -> Environment -> Add variable
   - Key: PORT
   - Value: 3000 (or whatever your config expects)
2. Or add it to netlify.toml in your repo so builds pick it up:

```toml
# netlify.toml
[build.environment]
PORT = "3000"
```

This will satisfy the requirement in vite.config.ts and allow the build to proceed.

Option B â€” Robust fix (recommended): make vite.config.ts only require PORT for dev serve, not for build
Edit artifacts/mockup-sandbox/vite.config.ts so it does not throw during production builds. Use the Vite config function parameter to distinguish commands. Example change:

```ts
// artifacts/mockup-sandbox/vite.config.ts
import { defineConfig } from 'vite';

export default defineConfig(({ command }) => {
  // Only require PORT when running the dev server (command === 'serve')
  if (command === 'serve' && !process.env.PORT) {
    throw new Error('PORT environment variable is required but was not provided.');
  }

  return {
    // ... your existing config
    server: {
      port: process.env.PORT ? Number(process.env.PORT) : undefined,
    },
  };
});
```

Why this helps: Netlify runs a build (command === 'build'), not the dev server, so the config will no longer throw during CI. This is the preferred approach because the build process should not depend on a dev-only environment variable.

Which to choose:
- If you control vite.config.ts, implement Option B to avoid build-time failures in CI.
- If you cannot change the repo quickly, use Option A to unblock the deploy immediately.
